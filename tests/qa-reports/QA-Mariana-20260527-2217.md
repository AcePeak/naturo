## Phase 4 — Scripter persona
This round's persona. Findings drove the two new issues:

- **#864** — Only `naturo click` accepts `--id eN`. All other 8 element-targeting commands (`get`, `type`, `press`, `find`, `drag`, `highlight`, `set`, `wait`) reject `--id` with Click's "No such option" exit=2. Issue #23 fixed `click` but never propagated. Each command uses a different flag (`-r/--ref`, `--on`, `--from-element`, positional `POSITIONAL_REFS`, etc.) — Scripter must memorize per-command names.
- **#865** — `naturo see -j` outputs a **bare element tree** on success (`{"id":"e1","role":"Window","children":[...]}`) but an **envelope** on error (`{"success": false, "error": {...}}`). All sibling commands (`list windows -j`, `list screens -j`, `config -j show`) consistently wrap success with `{"success": true, ...}`. Confirmed in source: `naturo/cli/core/_see.py:459` echoes `to_dict(tree)` directly. A naive `jq 'if .success then ...'` mishandles the success path.
- Click-level option validation (invalid `--backend`, invalid int for `--depth`, unknown flag, unknown subcommand) **does not respect** `-j`. Scripts get two error formats: exit=2 Click usage text vs exit=1 JSON envelope. Not filed separately — partially documented under #834 family; will fold into #865 if Scripter usage grows.
