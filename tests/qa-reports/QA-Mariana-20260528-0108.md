## Phase 4 — Simulated User: AI Agent Builder

Mindset: I'm wiring an AI agent that calls naturo via MCP, parsing JSON-RPC responses, and trusting `success: true`.

### Setup probe
- `naturo mcp start` boots cleanly on stdio.
- `initialize` (protocolVersion 2024-11-05) returns proper capabilities + serverInfo `{"name":"naturo","version":"1.26.0"}`.
- `tools/list` returns **64 tools** with `{name, description, inputSchema}` triples.

### What works
- `list_windows` returns real windows on the desktop (claude terminal, chrome).
- `list_monitors` returns `3840x2160`, `scale_factor=1.5`, `dpi=144`, primary `WinDisc`.
- `capture_window` (with `window_title`) correctly surfaces a structured error: `{"success": false, "error": {"code": "INTERNAL_ERROR", "message": "NaturoCoreError: capture_window: System/COM error"}}`.
- Pydantic-leak from #844 reproduces verbatim on bad args (`"1 validation error for clickArguments\nx\n  Input should be a valid integer..."`). No regression, no improvement.

### What's broken — new finding (P0)

`capture_screen` (with or without `output_path`):
- Response envelope says `"success": true`.
- Reports correct screen dims `3840×2160` with `scale_factor=1.5`, `dpi=144`.
- Returns a 24 KB PNG via `data_base64`.
- Decoded PNG opens in PIL as `(3840, 2160) RGB`, **extrema `((0,0), (0,0), (0,0))`** — every pixel black.
- With `output_path` argument, the same all-black PNG lands on disk.

CLI parallel call in the same session:
```
$ naturo capture --app notepad -o /tmp/x.png
Error: No interactive desktop in this session. ...
EXIT=1
```

So the CLI fails fast with `NO_DESKTOP_SESSION`; MCP synthesizes a fake-successful black PNG. This is the textbook silent-failure pattern SOUL.md flags as P0. Filed as **#868**.

### Lateral check (didn't widen the bug, but worth recording)
- `see_ui_tree` with no args → structured `NO_WINDOW` error (good).
- `app_inspect` (with wrong arg name `app_name` instead of `name`) → structured `INVALID_INPUT` (good — schema is honest; I just used the wrong key).

So the silent-failure surface is narrow but lethal: it's specifically `capture_screen` that doesn't propagate the session-binding error.

## Top 3 Risks

1. **MCP path has a silent-failure (#868, P0).** Any AI agent using `capture_screen` in a session that can't bind the desktop receives a fake-successful black PNG and will make wrong downstream decisions. Fix should be paired with #863 (shared session/UIPI root cause) — likely one missing precondition check at the screenshot backend entry point on the MCP path.
2. **#863 5th consecutive round.** Ship gate hasn't moved. Three workarounds still on the table (console session, accept unit-test evidence + manual smoke, cloud VM #860 day 22 unassigned).
3. **CLI/MCP contract asymmetry surfaced this round.** CLI propagates `NO_DESKTOP_SESSION` cleanly, MCP silently fakes success on one path (`capture_screen`) and properly errors on others (`capture_window`, `see_ui_tree`). The contract is inconsistent across tools. Suggest a single shared session-bind precondition check at the MCP tool boundary.
