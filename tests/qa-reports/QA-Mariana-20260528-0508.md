## Phase 4 ‚Äî Accessibility User simulation
Adopted the persona: keyboard-only, screen-reader-relying user trying to script naturo without ever touching a mouse.

| Observation | Verdict |
|------------|---------|
| `naturo see` exposes `role`, `name`, `bounds` in element tree | a11y-friendly ‚úì |
| `naturo press` supports every keyboard combo (`alt+f4`, `ctrl+a`, `tab --count 3`) | ‚úì |
| `naturo find --role Button --actionable` | role-based discovery ‚úì |
| `naturo get --role Edit --name Search -p value` | screen-reader-style read ‚úì |
| `naturo find` has `--role` but no `--name` flag (positional QUERY only) | **inconsistent with `get --role/--name`** ‚Äî friction for a11y users who script symmetrically across get/find |
| `naturo click --id` help text says "Automation element ID" but per #864 it accepts `eN` refs too | overloaded semantics ‚Äî confusing for a user who can't rely on visual context |
| `naturo -j --version` for env-probe | **plain text, not JSON** ‚Äî breaks a11y user's script that parses version via `json.parse` (filed as #874) |

**Filed during this persona**: #874 (`-j --version` / `-j --help` / `-j <unknown-cmd>` bypass JSON envelope).
