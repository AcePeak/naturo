**2. Click parse-error class systematically bypasses `-j` envelope (extended #872)**
Original #872 title: "unknown CLI flag emits plain Click usage text". Confirmed the bug is broader — six distinct Click parse-error categories all bypass `-j`:

| # | Category | Reproducer |
|---|----------|------------|
| 1 | Missing required arg | `naturo clipboard set -j` |
| 2 | Unknown flag | `naturo type --bogus-flag-xyz -j` |
| 3 | Too many positional args | `naturo wait 1 2 -j` |
| 4 | Invalid float type | `naturo wait abc -j` |
| 5 | Invalid integer type | `naturo see --hwnd abc -j` |
| 6 | Invalid Choice value | `naturo --log-level bogus see -j` |

All six emit plain Click banner on stderr (exit 2) instead of `{success: false, error: {...}}` on stdout. Suggested fix scope: one CliRunner-level catch handler converting `click.UsageError`/`NoSuchOption`/`BadOptionUsage`/`BadParameter` → standard envelope when `-j` is on the command line. Scope-expansion comment posted to #872; TC-0059 updated.

**3. `naturo clipboard set` lacks `--file`/stdin — asymmetric with `naturo type --file` (#888, P2)**
`naturo type` already accepts `--file PATH` for reading text from disk. `naturo clipboard set` accepts only positional `TEXT`. Three consequences:
- Hard cap at ~32KB via shell ARG_MAX (MSYS / PowerShell)
- Help example `naturo clipboard set "line1\nline2"` (advertised as "Multi-line text") stores `\n` *literally* — length 12, not 11. Help is misleading.
- For Skeptical Evaluator comparing to pywinauto (`pyperclip.copy(open('file').read())` is trivial Python), naturo has no equivalent CLI path

Both an ergonomics gap and a docs-accuracy gap. One-line internal fix: reuse the same `--file PATH` handler as `naturo type --file`. TC-0074 added.
