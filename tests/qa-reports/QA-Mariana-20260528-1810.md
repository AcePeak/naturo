## Phase 4 — Enterprise RPA Dev simulation (deep finding)

Designed and ran a realistic workflow gate: `naturo wait --gone "Loading dialog" --timeout 5 && naturo click "Save"`. In NO_DESKTOP_SESSION:

```
$ naturo wait --gone "Loading dialog" --timeout 0.5 -j && echo PROCEEDING && naturo click "Save" -j
{"success": true, "found": true, "wait_time": 0.016, "warnings": []}
PROCEEDING
{"success": false, "error": {"code": "NO_DESKTOP_SESSION", ...}}
```

`wait --gone`:
- Returns `success:true exit 0` in **0.236s** for a `--timeout 5` request (process startup time only; no polling done)
- An RPA workflow chained via `&&` proceeds to the next step on this false green light
- The failure surfaces 1-2 steps downstream at an unrelated command — agent logs show no correlation to "wait actually never ran"

Same architectural gap as #875/#878/#883 — desktop-required CLI surface bypasses the centralized guard described in epic #885. Filed as **#893 P1** and added to the #885 affected-surface matrix.

Cross-cutting: `wait --window` / `wait --element` (positive variants) at least correctly exit 1, but `-j` envelope omits the `error` object entirely. That's an envelope shape drift — added as scope-extension comment on #884.

## Top 3 Risks

1. **`wait --gone` silent success is the most agent-hostile bug in the cluster so far.** It explicitly violates the "post-action verify" claim in the README: a "wait" command that returns `wait_time:0.0` after a `--timeout 5` request — and reports success — is impossible to defend in any RPA SLA narrative. An agent that uses `wait --gone` as a gate before a destructive operation (close window with unsaved changes, submit a form, delete a file) will trigger that operation in a session where it has zero desktop binding. Promote #893 alongside #885 — they should ship together.
2. **The silent-failure surface has now grown to 6 CLI/MCP entrypoints** (`app windows`, `dialog detect`, `taskbar list`, `tray list`, `wait --gone`, MCP `capture_screen`, MCP `list_windows`). All have the same root cause: per-command guard placement instead of centralized middleware. #885's "central guard + CI gate" plan remains the right answer and is more urgent — every new feature added without it adds another row to this matrix.
3. **The 5 SendInput-blocked status:done remain unverified ~25 hours after the restructured ship gate** (per R137 risk #3, still unaddressed). v0.3.2 release date drifts daily. Without a console-session QA run, even closing #885 cannot ship v0.3.2. Escalate to Ace.
