## Phase 4 ‚Äî Skeptical Evaluator verdict
**The pitch is strong; the trust model is not.** naturo's differentiators over pyautogui (0.9.54,
installed) and pywinauto are real on paper: a CLI + an MCP server + JSON output + element-tree
`see`/`click eN` (vs pyautogui's blind coordinate clicks, vs pywinauto's library-only/no-CLI/no-MCP).
For an agent builder, "spawn `naturo mcp start` and call tools" is genuinely more ergonomic than
embedding pywinauto.

**But the deciding question for a skeptic is "can I trust the output?" ‚Äî and today the answer is no:**
- The README markets ‚úÖ "AI Agent Ready: JSON output" and ‚úÖ "Post-Action Verify" (#887/TC-0075),
  yet the agent-facing contract is exactly where it leaks: MCP `list_apps` returns `success:true`
  with plausible-but-context-invalid data while the CLI correctly refuses. An agent that branches on
  `success`/`isError` gets silently lied to ‚Äî the worst failure mode for autonomous use.
- JSON envelope is not uniform (`selector list` ‚Üí `{}`, `-j --version` bypasses, `app windows`
  success-shape vs `see` error-shape), so a script can't write one parser.
- `serverInfo.version` reporting the SDK version means a skeptic can't even tell which naturo they
  pinned.

**Adoption call:** I would *prototype* with naturo for the CLI/MCP ergonomics, but I would **not yet
ship an unattended agent on it** until the #885 guard cluster + the JSON-envelope drift cluster
(#874/#876/#882/#884) are closed. That single fix (centralized `require_desktop_session` + one
envelope schema) flips the verdict ‚Äî it's a small, high-leverage gate, not a feature backlog.
This is consistent with #887; no new issue filed (would duplicate).
